The module registers ordered `#backend/*` aliases (for both Vite and Nitro) so you never spell out `_generated`:

```ts
import { api } from '#backend/api'
import { query, mutation } from '#backend/server'
import type { Doc, Id } from '#backend/dataModel'
```

| Alias | Resolves to |
|---|---|
| `#backend/api` | `_generated/api` |
| `#backend/server` | `_generated/server` |
| `#backend/dataModel` | `_generated/dataModel` |
| `#backend/_generated` | `_generated` |
| `#backend` | your functions directory |

::callout{icon="i-lucide-info"}
Order matters: the specific generated-module aliases are registered **before** the catch-all `#backend`, because the alias resolver is first-match-wins and `#backend` is a prefix of `#backend/api`.
::

The functions directory resolves in order: the `functions` field in `convex.json` (normalized — leading `./` and trailing slashes stripped), then an existing `convex/` directory, then an existing `backend/` directory, and finally the `convex/` default. If `convex.json` fails to parse, the module warns and falls through to the directory checks.

Beyond `#backend/*`, the module registers one more alias: `#convex/auth-client`, which the Better Auth runtime uses to resolve your auth-client module — see the [configuration reference](/getting-started/configuration#import-aliases) and [bring your own auth client](/guide/authentication#bring-your-own-auth-client).

You often don't need `#backend/api` at all: a generated plugin auto-provides the `api` app-wide, so namespace-reading composables and components (`useBilling()`, `<CheckoutLink>`, …) work with zero arguments. Before `convex dev` has emitted `_generated/`, the plugin is a no-op and those features degrade gracefully.
