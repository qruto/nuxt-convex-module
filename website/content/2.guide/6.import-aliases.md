The module registers ordered `#convex/*` aliases (for both Vite and Nitro) so you never spell out `_generated`:

```ts
import { api } from '#convex/api'
import { query, mutation } from '#convex/server'
import type { Doc, Id } from '#convex/dataModel'
```

| Alias | Resolves to |
|---|---|
| `#convex/api` | `_generated/api` |
| `#convex/server` | `_generated/server` |
| `#convex/dataModel` | `_generated/dataModel` |
| `#convex/_generated` | `_generated` |
| `#convex` | your functions directory |

::callout{icon="i-lucide-info"}
Order matters: the specific generated-module aliases are registered **before** the catch-all `#convex`, because the alias resolver is first-match-wins and `#convex` is a prefix of `#convex/api`.
::

The functions directory resolves in order: the `functions` field in `convex.json` (normalized — leading `./` and trailing slashes stripped), otherwise the `convex/` default. If `convex.json` fails to parse, the module warns and falls back to the `convex/` default.

Beyond `#convex/*`, the module registers one more alias: `#convex/auth-client`, which the Better Auth runtime uses to resolve your auth-client module — see the [configuration reference](/getting-started/configuration#import-aliases) and [bring your own auth client](/guide/authentication#bring-your-own-auth-client).

You often don't need `#convex/api` at all: a generated plugin auto-provides the `api` app-wide, so namespace-reading composables and components (`useBilling()`, `<CheckoutLink>`, …) work with zero arguments. Before `convex dev` has emitted `_generated/`, the plugin is a no-op and those features degrade gracefully.
