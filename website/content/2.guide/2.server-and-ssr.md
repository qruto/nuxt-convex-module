These server utilities mirror `convex/nextjs`. They are auto-imported in Nitro server code — server routes, API handlers, and server middleware. In app-side code (pages, components, `useAsyncData`), even when it runs during SSR, import them explicitly from `nuxt-convex-module/server`. All of them read the deployment URL from `NUXT_PUBLIC_CONVEX_URL` (`npx convex dev` sets it during local development).

## One-shot calls

```ts [server/api/messages.get.ts]
import { api } from '#convex/api'

export default defineEventHandler(async () => {
  return await fetchQuery(api.messages.list, {})
})
```

`fetchQuery`, `fetchMutation`, and `fetchAction` run a single Convex call over HTTP (no subscription). They (and `preloadQuery`) accept a trailing options object: `token` — a JWT-encoded OpenID Connect token, the manual path to authenticated calls without Better Auth — plus `url` to override the deployment URL and `skipConvexDeploymentUrlCheck` for self-hosted deployments:

```ts
await fetchMutation(api.messages.send, { body }, { token })
```

Full signatures live in the [API reference](/api-reference/reference/server).

## `useAsyncQuery` — SSR data the Nuxt way

`useAsyncQuery` is the one-stop composable for pages that should render with data on the server *and* stay live on the client. It models Nuxt's `useAsyncData`: during SSR the query runs over HTTP and the result is embedded in the Nuxt payload; after hydration the composable upgrades to the WebSocket subscription — no loading flash, no client refetch.

```vue [pages/tasks.vue]
<script setup lang="ts">
import { api } from '#convex/api'

const { data: tasks, status, error } = useAsyncQuery(api.tasks.list, { completed: false })
</script>

<template>
  <div v-if="status === 'pending'">Loading…</div>
  <div v-else-if="error">{{ error.message }}</div>
  <TaskList v-else :tasks="tasks" />
</template>
```

It returns Nuxt's familiar `{ data, error, status, refresh }` refs and is awaitable (`await useAsyncQuery(...)` blocks like `useAsyncData`). Compared to `useQuery` — which renders `undefined` during SSR and fetches on the client — `useAsyncQuery` puts real data in the server-rendered HTML. Compared to `preloadQuery` + `usePreloadedQuery`, there's no server route or prop-threading.

Options (all optional, third argument):

- `server` (default `true`) — fetch during SSR; `false` fetches on the client only.
- `lazy` (default `false`) — don't block client-side navigation on the initial fetch.
- `live` (default `true`) — upgrade to the live subscription on the client. With `live: false` you get SSR + hydration *without* a WebSocket — `refresh()` is then the way to get fresh data. With `live: true`, `refresh()` is mainly for retrying after an error; the server pushes updates anyway.
- `key` — the payload/dedup key; defaults to the query name + initial args.
- `token` — a JWT (or a function resolving one) to authenticate the SSR fetch. With the Better Auth integration enabled, the request's prefetched token is used automatically.

Pass `'skip'` as args (or a reactive getter returning it) to pause the query. Reactive args are honored by the live subscription; the payload entry is keyed on the initial args and only matters for first paint.

## Preloading for SSR

Preload on the server, hydrate on the client without a flash or a refetch. The `Preloaded` payload is a plain JSON-serializable object designed to cross the server/client boundary — return it from a server route (or `useAsyncData`), then hand it to the consuming component, for example as a prop:

```ts [server/api/messages.get.ts]
import { api } from '#convex/api'

export default defineEventHandler(async () => {
  return await preloadQuery(api.messages.list, {})
})
```

```vue [components/Messages.vue]
<script setup lang="ts">
import type { api } from '#convex/api'
import type { Preloaded } from 'nuxt-convex-module/client'

const props = defineProps<{
  preloaded: Preloaded<typeof api.messages.list>
}>()

const messages = usePreloadedQuery(props.preloaded)
</script>
```

`usePreloadedQuery` renders the preloaded value identically on server and initial client, then takes over as a live subscription — so there's no hydration mismatch. It returns a computed ref: templates unwrap it, script code reads `messages.value`. The payload argument also accepts a ref, computed, or getter — a fresh payload re-subscribes with the new query and args.

`preloadedQueryResult(preloaded)` extracts just the value when you don't need the live handoff.

## Authenticated server calls

With the Better Auth integration installed, `convexAuth(event)` returns a request-scoped, authenticated helper service (`getToken`, `isAuthenticated`, `fetchAuthQuery`/`fetchAuthMutation`/`fetchAuthAction`, `preloadAuthQuery`, and `handler` — the Better Auth route proxy). It's also auto-imported under its upstream-style name, `convexBetterAuthNuxt`:

```ts [server/api/me.get.ts]
import { api } from '#convex/api'

export default defineEventHandler(async (event) => {
  const { fetchAuthQuery, isAuthenticated } = convexAuth(event)
  if (!await isAuthenticated()) {
    throw createError({ statusCode: 401 })
  }
  return await fetchAuthQuery(api.auth.getAuthUser)
})
```

An optional second argument takes Better Auth's `GetTokenOptions` (e.g. `jwtCache`) plus a `convexSiteUrl` override — it defaults to `NUXT_PUBLIC_CONVEX_SITE_URL`. See the [API reference](/api-reference/reference/better-auth/nuxt/server) for the full service shape.

::callout{icon="i-lucide-triangle-alert"}
Authenticated SSR responses embed per-user secrets in the HTML: the server plugin serializes the signed-in user's Convex JWT into the Nuxt payload (`window.__NUXT__`) so the client can take over without a refetch — mirroring upstream's `initialToken` handoff — and `preloadAuthQuery` data rides the same payload. Never let a CDN, reverse proxy, or any shared cache store these pages keyed only on the URL: one user's HTML (token included) would be served to another. Serve every route that renders for a signed-in user with `Cache-Control: private, no-store`.
::
