Convenience composables over Convex's file storage. All auto-imported.

## `useUpload()`

Upload a single file. Pass your `generateUploadUrl` mutation reference ‚Äî the composable calls it to obtain the one-time upload URL, `POST`s the file, and reads back the storage id:

```vue
<script setup lang="ts">
import { api } from '#convex/api'

const { upload, progress, isUploading, error } = useUpload(api.files.generateUploadUrl)

async function onChange(e: Event) {
  const file = (e.target as HTMLInputElement).files?.[0]
  if (file) {
    const storageId = await upload(file)
    if (storageId === null) console.error(error.value)
  }
}
</script>
```

`upload()` never throws: it resolves to the new `StorageId`, or to `null` when the upload fails (or runs during SSR ‚Äî uploads are client-only), setting the `error` ref with the reason.

The full return surface is `{ upload, isUploading, progress, error, storageId, cancel, reset }` ‚Äî `progress` is a `0`‚Äì`1` ref, `storageId` holds the last successful id, `cancel()` aborts the in-flight upload, and `reset()` clears `progress`, `error`, and `storageId`.

An optional second argument takes callbacks ‚Äî `onSuccess(storageId, file)`, `onProgress(fraction)`, and `onError(error, file)`: `useUpload(api.files.generateUploadUrl, { onSuccess: storageId => saveImage({ storageId }) })`.

## `useUploadQueue()`

Manage several uploads at once, each with its own status and progress (a deeply reactive queue):

```ts
const { items, enqueue, progress, isUploading } = useUploadQueue(
  api.files.generateUploadUrl,
  { concurrency: 4, onItemSuccess: storageId => saveImage({ storageId }) },
)
```

`enqueue` accepts a single `Blob`, a `Blob[]`, or a `FileList` straight from a multi-file `<input>` ‚Äî `enqueue((event.target as HTMLInputElement).files)` ‚Äî and uploading starts automatically, at most `concurrency` at a time (default `3`). Each entry in `items` carries `{ id, file, status, progress, storageId, error }` with `status` moving through `'pending' | 'uploading' | 'success' | 'error'`.

Besides `items` and `enqueue`, the queue returns aggregate `progress` and `isUploading` refs, `pendingCount`/`activeCount`, and `cancel()` (abort in-flight uploads), `remove(id)`, and `clear()`. Options also take `onItemError(error, item)` and `onComplete(items)` ‚Äî called once every queued item has settled. Full signatures live in the [API reference](/api-reference/reference/client#useuploadqueue).

## `useStorageUrl()`

Resolve a stored file's served URL (reactive to the id):

```ts
const { storageId } = useUpload(api.files.generateUploadUrl)
const url = useStorageUrl(api.files.getUrl, storageId)
```

The id may be a plain value, ref, computed, or getter; while it is `null`/`undefined` the query is skipped automatically ‚Äî no `'skip'` juggling. The return is a `ComputedRef<string | null | undefined>`: `undefined` while loading or skipped, `null` when the file no longer exists, the URL once resolved. Templates unwrap it (`<img v-if="url" :src="url">`); in script use `url.value`.

::callout{icon="i-lucide-shield"}
The bundled CSP (shipped via the `nuxt-security` module dependency) allows Convex-served storage on `img-src` / `media-src` ‚Äî in `nuxt dev` as well as production builds. This requires the Convex URL to be known at build time and is skipped if you set `security: false` or disable `contentSecurityPolicy` yourself.
::
