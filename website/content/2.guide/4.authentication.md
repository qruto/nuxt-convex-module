## Provider-agnostic core

The core ships the generic Convex auth plumbing — a port of `convex/react`'s `ConvexProviderWithAuth`:

- `provideConvexAuth({ client, useAuth })` — wire any auth provider's loading/authenticated flags + token fetcher into the Convex client. Re-authentication has two triggers: pass `fetchAccessToken` as a `Ref` whose identity changes, or return the additive `authVersion` key (`MaybeRefOrGetter<unknown>`) — when either changes, auth state transitions back to loading and the token is fetched again. The bundled adapters rely on this (Clerk keys `authVersion` off the active org id + role).
- `useConvexAuth()` — read `{ isLoading, isAuthenticated, isRefreshing }` anywhere. Each field is a `ComputedRef<boolean>`, so the upstream destructuring idiom stays reactive:

  ```ts
  const { isLoading, isAuthenticated } = useConvexAuth()
  // isAuthenticated.value in script, plain `isAuthenticated` in templates
  ```

- `<Authenticated>`, `<Unauthenticated>`, `<AuthLoading>`, `<AuthRefreshing>` — render by auth state.

```vue
<template>
  <Authenticated>Welcome back</Authenticated>
  <Unauthenticated>Please sign in</Unauthenticated>
  <AuthLoading>Checking…</AuthLoading>
  <AuthRefreshing>Refreshing your session…</AuthRefreshing>
</template>
```

`isRefreshing` (and `<AuthRefreshing>`) is `true` only while an already-authenticated session is re-fetching a token the server rejected — the socket is paused meanwhile. Routine background token rotation does not trigger it.

## Opt-in: Clerk

Install `@clerk/vue` and the Clerk adapter wires itself up — a Vue port of `convex/react-clerk`. Wrap your app with Clerk's plugin, then call the composable (or use the component) in a top-level component:

```bash
npm i @clerk/vue
```

```vue
<script setup lang="ts">
// A Vue port of `<ConvexProviderWithClerk>`.
provideConvexAuthFromClerk()
</script>

<template>
  <Authenticated>Signed in with Clerk</Authenticated>
  <Unauthenticated>Sign in</Unauthenticated>
</template>
```

The adapter reads Clerk's `useAuth()`. When Clerk's native Convex integration is active (the session token's `aud` claim is `convex`), it fetches the session token directly; otherwise it falls back to the `convex` JWT template. Either way the token is re-fetched when the active organization *or* the user's org role changes. Drop-in component form: `<ConvexProviderWithClerk>`. Types via `@qruto/nuxt-convex/clerk/vue`.

## Opt-in: Auth0

Install `@auth0/auth0-vue` and the Auth0 adapter wires itself up — a Vue port of `convex/react-auth0`. Wrap your app with the Auth0 plugin, then:

```bash
npm i @auth0/auth0-vue
```

```vue
<script setup lang="ts">
// A Vue port of `<ConvexProviderWithAuth0>`.
provideConvexAuthFromAuth0()
</script>
```

The adapter reads Auth0's `useAuth0()` and authenticates Convex with the Auth0 `id_token`. Drop-in component form: `<ConvexProviderWithAuth0>`. Types via `@qruto/nuxt-convex/auth0/vue`.

## Opt-in: Better Auth

Install `@convex-dev/better-auth` and the integration wires itself up — a Vue/Nuxt port of `@convex-dev/better-auth`'s `react` + `nextjs` integration:

```bash
npm i @convex-dev/better-auth better-auth @better-auth/passkey
```

You now get:

- **`useAuth()`** (auto-imported) — the Better Auth session, plus `signOut`, OTP sign-in, passkeys, and more.
- **`/api/auth/**`** — a registered same-origin proxy route to your Convex deployment's Better Auth endpoints (route configurable via `convex.authRoute`).
- **`auth` route middleware** — opt a page in with `definePageMeta({ middleware: 'auth' })`. Unauthenticated visitors are redirected to `/login` on both server and client; the path is not configurable, so make sure a `/login` route exists (or write your own middleware for a different target).
- **`usePreloadedAuthQuery()`** (auto-imported) — hydrate an SSR-preloaded authenticated query.
- **`backendAuth(event)`** (auto-imported in Nitro server routes; also under its upstream name `convexBetterAuthNuxt`) — a request-scoped authenticated client for server routes.

```vue
<script setup lang="ts">
const { session, signInWithOtp, signOut } = useAuth()
</script>
```

### Bring your own auth client

By default the integration ships a Better Auth client with `convexClient` + `emailOTPClient` + `passkeyClient`. This mirrors `@convex-dev/better-auth`'s React integration, where you author the `authClient` and pass it to `ConvexBetterAuthProvider` — here you point the module at your own module instead. The aliased client backs `useAuth()`, the client plugin (session wiring and cross-domain token consumption), and the `AuthClient` type of `<AuthBoundary>`'s `authClient` prop — the boundary still takes the client instance as a required prop. The server SSR plugin doesn't touch it at all: it prefetches the token over HTTP via `backendAuth(event).getToken()`.

```ts
// app/convex-auth-client.ts
import { createAuthClient } from 'better-auth/vue'
import { convexClient, crossDomainClient } from '@qruto/nuxt-convex/better-auth/vue'

export const authClient = createAuthClient({
  plugins: [convexClient(), crossDomainClient()], // add the plugins you need
})
export type AuthClient = typeof authClient
```

```ts
// nuxt.config.ts
export default defineNuxtConfig({
  convex: { betterAuth: { authClient: './app/convex-auth-client' } },
})
```

Adding `crossDomainClient()` enables **cross-domain auth**: on landing, `consumeCrossDomainOneTimeToken()` (called automatically by the client plugin) exchanges a `?ott=…` one-time token for a session. Leave it out for same-origin apps — the default client keeps cookies on the same origin.

::callout{icon="i-lucide-triangle-alert"}
The `useAuth()` conveniences map to specific client plugins: `sendOtp` / `signInWithOtp` need `emailOTPClient()`, and `signInWithPasskey` / `registerPasskey` need `passkeyClient()`. The bundled default client includes both — a custom client that omits them (like the example above) makes those wrappers fail at runtime.
::

### `<AuthBoundary>` (experimental)

Redirect unauthenticated users reactively, and clear an invalid session before redirecting:

```vue
<script setup lang="ts">
import { authClient } from '@qruto/nuxt-convex/better-auth/vue'
import { api } from '#backend/api'
import { isAuthError } from '~/utils/auth' // your own predicate
</script>

<template>
  <AuthBoundary
    :auth-client="authClient"
    :get-auth-user-fn="api.auth.getAuthUser"
    :is-auth-error="isAuthError"
    :on-unauth="() => navigateTo('/login')"
  >
    <slot />
  </AuthBoundary>
</template>
```

Only the `<AuthBoundary>` component itself is auto-registered — `authClient` must be imported (the export resolves through the `#convex/auth-client` alias to your configured client, so it's the same instance `useAuth().client` returns), and `isAuthError` is your own error predicate. `getAuthUserFn` expects the query from `authComponent.clientApi()` in your Convex backend.

::callout{icon="i-lucide-info"}
Better Auth is **not** a default — exactly as in Convex's upstream integration, where it's a separate package. Install it only if you need it.
::
