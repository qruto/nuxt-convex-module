All of these are auto-imported. `useQuery`, `useQueries`, `useMutation`, `useAction`, `usePaginatedQuery`, `useUpload`, `useUploadQueue`, and `useStorageUrl` also have `useConvex*`-prefixed aliases to avoid name clashes.

## `useQuery()`

Returns a `ComputedRef<T | undefined>` — `undefined` while loading — and **throws** query errors (caught by an `errorCaptured` boundary, mirroring React's render-throw). Args are reactive; pass `'skip'` to pause.

```vue
<script setup lang="ts">
import { api } from '#backend/api'

const messages = useQuery(api.messages.list, {})

const profile = useQuery(
  api.users.get,
  computed(() => userId.value ? { userId: userId.value } : 'skip'),
)
</script>
```

### `useQuery_experimental()`

The convex 1.37 result/error split — takes an options object (`{ query, args, throwOnError? }`) and returns a `{ status }` union instead of throwing:

```ts
const result = useQuery_experimental({ query: api.messages.list, args: {} })
// result.value: { status: 'pending' }
//             | { status: 'success', data }
//             | { status: 'error', error }
```

Pass `throwOnError: true` to opt back into throwing to `errorCaptured` — the `'error'` variant is then excluded from the type.

## `useQueries()`

Load a dynamic number of live queries from a single composable call — without violating the rules of composables:

```ts
// Pass a getter (or ref/computed) to make the whole map reactive — per-query
// args are plain values, not refs.
const results = useQueries(() => ({
  messages: { query: api.messages.list, args: {} },
  profile: { query: api.users.get, args: { userId: userId.value } },
}))
```

The return is a `ShallowRef` of a record with the same keys as the input. Each value is the query result, `undefined` while loading, or an `Error` instance if the query threw — unlike `useQuery`, errors are **not** thrown, so check `instanceof Error` yourself. In script, read `results.value.messages`; templates unwrap the ref.

## `useMutation()` / `useAction()`

```ts
const send = useMutation(api.messages.send)
await send({ body: 'hi' })

const checkout = useAction(api.payments.checkout)
```

Both return a stable plain function (no `.value` needed). `useMutation` also supports [optimistic updates](#optimistic-updates).

## Optimistic updates

Chain `.withOptimisticUpdate()` onto the function returned by `useMutation` to update local query results before the server responds. The handler must be synchronous, receives the client's local query store and the mutation args, and is rolled back automatically once the mutation completes and the real results sync. It returns a **new** `VueMutation` — and throws if the mutation already has an update configured, so call it once per `useMutation`, not per invocation.

```ts
import { api } from '#backend/api'

const toggle = useMutation(api.tasks.toggle).withOptimisticUpdate(
  (localStore, args) => {
    const current = localStore.getQuery(api.tasks.get, { id: args.id })
    if (current !== undefined) {
      localStore.setQuery(api.tasks.get, { id: args.id }, {
        ...current,
        completed: !current.completed,
      })
    }
  },
)
```

### Paginated helpers

Updating a paginated query by hand means finding the right page. Four helpers (same names and shapes as `convex/react`, importable from `@qruto/nuxt-convex/vue`) do it for you:

| Helper | What it does |
| --- | --- |
| `insertAtTop({ paginatedQuery, argsToMatch?, localQueryStore, item })` | Prepends `item` to the first page. No-op until the first page has loaded. |
| `insertAtBottomIfLoaded({ paginatedQuery, argsToMatch?, localQueryStore, item })` | Appends `item` to the final page — only if the last page has loaded. |
| `insertAtPosition({ paginatedQuery, argsToMatch?, sortOrder, sortKeyFromItem, localQueryStore, item })` | Inserts `item` at its sorted position across the loaded pages. |
| `optimisticallyUpdateValueInPaginatedQuery(localStore, query, args, updateValue)` | Maps `updateValue` over every loaded item of the paginated query matching `args`. |

For a chat UI backed by `usePaginatedQuery(api.messages.list, ...)`, show the sent message instantly:

```ts
import { insertAtTop } from '@qruto/nuxt-convex/vue'
import { api } from '#backend/api'
import type { Id } from '#backend/dataModel'

const sendMessage = useMutation(api.messages.send).withOptimisticUpdate(
  (localStore, args) => {
    insertAtTop({
      paginatedQuery: api.messages.list,
      argsToMatch: { channel: args.channel },
      localQueryStore: localStore,
      item: {
        _id: crypto.randomUUID() as Id<'messages'>,
        _creationTime: Date.now(),
        channel: args.channel,
        body: args.body,
      },
    })
  },
)

await sendMessage({ channel: '#general', body: 'hi' })
```

The temporary `_id` is replaced by the real document when the mutation's results sync.

## `usePaginatedQuery()`

Cursor pagination with a manual page-management engine:

```ts
const { results, status, isLoading, loadMore } = usePaginatedQuery(
  api.messages.list,
  {},
  { initialNumItems: 20 },
)
```

`results`, `status`, and `isLoading` are `ComputedRef`s — templates unwrap them, script needs `.value` (e.g. `status.value === 'CanLoadMore'`). `status` is `'LoadingFirstPage' | 'LoadingMore' | 'CanLoadMore' | 'Exhausted'`. `loadMore(numItems)` is a stable function that only fetches when status is `'CanLoadMore'`. Like `useQuery`, reading any of the fields throws the query error to the nearest `errorCaptured` boundary.

### `usePaginatedQuery_experimental()`

A dual-overload variant: the positional form matches `usePaginatedQuery` above, while the object form takes a single (optionally reactive) options object — `{ query, args, initialNumItems, throwOnError? }` — and returns one `ComputedRef` of a lowercase-status union:

```ts
const page = usePaginatedQuery_experimental({
  query: api.messages.list,
  args: {},
  initialNumItems: 20,
})
// page.value: { data, status: 'pending' | 'success' | 'error',
//               canLoadMore, isLoading, error, loadMore }
```

On error (without `throwOnError`), `data` is an empty array — never the previously loaded pages.
