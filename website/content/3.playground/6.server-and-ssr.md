---
title: Server & SSR
description: Preload a query on the server with preloadQuery, hydrate it flicker-free with usePreloadedQuery, and contrast it with a one-shot fetchQuery.
---

This demo is rendered from data fetched **on the server**: a Nitro route runs
`preloadQuery` (and a one-shot `fetchQuery` for the count), the component picks
the payload up with `useFetch`, and `usePreloadedQuery` hydrates it without a
flash — then keeps it live over the WebSocket.

::playground-preload
::

```ts [server/api/playground/preload-messages.get.ts]
import { api } from '#backend/api'

// preloadQuery and fetchQuery are auto-imported in Nitro server code
export default defineEventHandler(async () => {
  const [preloaded, count] = await Promise.all([
    preloadQuery(api.messages.list, {}), // Preloaded payload for the client
    fetchQuery(api.messages.count, {}), // plain one-shot value
  ])
  return { preloaded, count }
})
```

```vue
<script setup lang="ts">
import type { api } from '#backend/api'
import type { Preloaded } from 'nuxt-convex-kit/vue'

// Outer component: carry the payload across the server/client boundary
const { data } = await useFetch('/api/playground/preload-messages')

// Inner component (gated with v-if="data" and given the payload as a prop):
const props = defineProps<{ preloaded: Preloaded<typeof api.messages.list> }>()
const messages = usePreloadedQuery(() => props.preloaded)
</script>
```

`preloadQuery` runs the query on the server — it accepts a `token`, so
authenticated preloads work too — and returns a JSON-serializable `Preloaded`
payload. `usePreloadedQuery` renders that payload identically on the server and
the first client paint (no hydration flicker, no refetch), then seamlessly
takes over as a live subscription: send a message and the list updates.
`fetchQuery` is a one-shot HTTP read with no subscription — that's why the
count in the demo stays frozen at its render-time value.
